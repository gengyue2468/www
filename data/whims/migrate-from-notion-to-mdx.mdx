在六月底到七月中下旬及至八月上旬（前几天吧），我一直在探索 Notion API 与 Next.js 结合起来制作个人页面。
通过 Notion 提供的数据库存储数据，然后通过 API 获取数据，这似乎是一个完美的无可挑剔的方案。

这个方案的优点如下：

1. Notion 可以随写随保存，内容可以自动同步到网站，无需重复部署
2. Notion 提供了丰富的块级元素，可以及其直观地写博文，而无需面对抽象的 Markdown 语言
3. Notion 提供储存，无需占用 Vercel 服务器带宽

值得注意，这个方案其实有可取之处的，但为什么最终我还是选择了迁移呢？这源于这套方案的几个缺点：

1. Notion 服务器在国内响应缓慢，页面加载过慢，容易造成累计布局偏移
2. Notion API 的请求会占用大量的 Vercel 免费额度（由于请求量巨大）
3. Notion 的块级元素在 next-mdx-remote 处理时容易出现问题
4. 由于 Notion API 请求需要 Edge Network，几乎无法找到除 Vercel 之外的免费解决方案

> 注意：我曾经尝试过使用 Tencent EdgeOne 提供的 Pages 服务来部署，但是参阅文档之后发现现阶段 EdgeOne 仅支持静态导出的 Next.js 项目，所以这个方案被否定

当然，使用`getStaticProps`有它本身的优点：

1. 更快速的加载体验
2. 更低的服务器负载
3. 更小的累计布局偏移
4. 可用 Vercel 提供的静态缓存

于是现如今，我回到了传统的 Markdown 写作之中，生成静态网页部署在 Vercel 上，我曾经尝试过使用 EdgeOne 免费套餐进行边缘加速，但是 Ping 了一下感觉非中国大陆区域的速度还赶不上 Vercel，
并且域名还时不时发生 HSTS 错误，所以无奈放弃（有待测试）。总之，Notion 是一个很不错的笔记软件，但是限于目前上述讲述的这些原因，我不得不放弃 Notion API，回归传统。这就是我的问题的答案。
